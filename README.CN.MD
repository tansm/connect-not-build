# 连接而不是构建到AI

## 背景

在2024~2025年期间，软件企业都花费了很多的精力去开发各种 AI Agent，也取得了很大的成果，在日新月异的AI大时代也算赶上了浪潮。现在，我们已经有了成果，需要回顾过去，展望未来，探讨软件企业下一步的发展路线。

回顾过去，我们看到：

- 研发成本高昂，主要是因为AI Agent SDK 早期不成熟，技术迭代也非常快，导致团队中途更换SDK，或仍然使用很早期的版本，进行了大量的定制，导致无法追赶新版本SDK。

- 产品功能不一致，各模块团队各自开发Agent，导致最终用户的体验不一致，比如某些Agent可能有记忆，有些Agent又没有，或者两个Agent的记忆不互通。

- 产品特性追不上快速的节奏，团队上周加班加点终于推出了记忆功能，可这周客户就问：其他产品都有子Agent协作，为什么你的产品还没有。

- 模块之间缺少协作能力，大型的软件包含众多的模块，财务的AI只能回答财务问题，要下钻到供应链分析，即使用户有权限，也需要切换到供应链的AI，又要复述一遍。

- 产品也缺少外部协作的能力，用户期望使用诸如 Codex, Claude cwork 这样的通用 AI 工具，从大型ERP软件中提取数据，然后用AI 编写复杂的 python 清洗逻辑，再让AI使用Excel清晰展现结果，经过会议讨论后，让AI重新将Excel的评审结果导入到财务预算模块。

## 分析原因

分析目前AI的工作模式和我们之前的工作内容，我们发现，我们主要做了两部分的工作为AI提供能力，一部分是AI通用的能力，包括但不限于 记忆，限流，权限，SKILL支持，多Agent等，另外一部分才是软件企业自己独特的能力的“包装”，即让AI能操作特定此软件领域的能力，这些能力都是原始软件本身的能力，我们只是包装成AI **能感知，能决策，能行动**。

关于AI通用的能力，我们首先的反应是：各产品线，模块应该统一AI Agent的底层平台，减少内耗。

那么特定领域的能力如何包装呢？这些能力“暴露” 给统一的AI底层平台，它耦合了特定的技术平台，可能基于 LangChain 的python 生态，也可能是 Microsoft Agent Framework 的.net平台。 这很像当年企业级软件开发时赌上 JAVA 的 Spring 生态，还是 .net 的 ASP.NET 生态。

我们需要找到一种 松耦合，广泛支持的标准，去包装特定领域的能力，比如 MCP, SKILL 等标准，让AI平台 **连接** 到软件的特定领域， 从而获得以下优势：

- 不再定制通用的AI平台，而是充分连接更优秀的AI平台，砍掉开发通用AI平台的成本；

- 因为是连接，天然就享受了最新的 AI 特性，大厂AI产品今天发布新特性，你的客户立即就能享受，而不是等你跟进；

- 同样因为是连接，所以功能是一致的，记忆是相通的，内部和外部的数据是相通的，AI很轻松的在各个软件之间协作。

MCP（模型上下文协议）是一种用于将人工智能应用与外部系统相连的开源标准。https://modelcontextprotocol.io/
Agent Skills 是一种轻量级、开放式的架构，可用于通过各种专业知识和工作流程来增强人工智能代理的功能。https://agentskills.io/
Agent2Agent（A2A）协议是一项开放标准，旨在实现人工智能代理之间的无缝通信与协作。 https://a2a-protocol.org/

## 方案

为了让 AI Agent 能够进入你的特定领域，我们需要向 AI 暴露以下的能力：

### **感知**

假想你现在需要组装一个宜家的家具， 你是不是先要看到这些木材，螺丝，工具，说明书等等，映射到AI与企业软件上，就是要让AI 感知到当前环境，包括但不限于：有哪些模块，工具，技能，更深的专业知识库在哪里找等。

感知还包括，“当前”信息，比如你停留在公司的协作聊天界面中，问AI：总结刚才大家都在聊啥。 为实现这个功能你必须让AI能感知到“什么是当前的聊天”，这种当前信息的感知还有很多，当前用户，当前的项目，当前操作的界面。

这个在 MCP 上对应的是 resources ， 但目前 MCP 中 resouces 是面向 AI 主机，AI 不能直接阅读，官方的建议是MCP提供特有的读取资源工具，返回的结果标记这是一个[资源](https://modelcontextprotocol.io/specification/2025-11-25/server/tools#resource-links)。

MCP 支持返回 instructions 向 AI [介绍自己](https://modelcontextprotocol.io/specification/2025-06-18/schema#initializeresult)，利用这个机制让 AI 快速的感知软件的特定领域。

MCP 的资源一样可以用来暴露“当前信息”，且还支持通知 Host 资源发生变化，这样当用户从一个聊天组切换到另外一个聊天组，也能及时更新。但你仍然需要使用 tools 暴露当前信息，AI 默认是不能直接访问到资源的。

### **技能**

继续宜家的比喻，你看到了所有的材料和工具，也看到了说明书，但你不会用电钻，不知道什么时候该用哪种螺丝，这时候你需要的是"技能"——如何正确地完成某个具体步骤的知识。映射到AI与企业软件上，技能就是告诉AI：在这个特定领域中，完成某类任务的标准流程、约束条件和最佳实践是什么。

技能与工具的区别在于：工具是原子化的操作能力（查询一条数据，提交一个表单），而技能是将多个工具编排起来完成一个有业务含义的任务的知识。例如"创建一笔采购订单"不是一个工具调用，而是一系列步骤：校验供应商资质、检查预算余额、填充默认税率、生成审批流、通知相关人。AI 需要知道这些步骤的顺序、前置条件和异常处理策略，这就是技能。

技能的表达方式有多种选择。Agent Skills 规范提供了一种标准化的描述格式，包含输入输出的 schema、执行步骤、前置条件等元数据，AI 可以据此理解并执行复杂的多步任务。

技能的另一个关键特征是可组合性。一个高层级的技能可以引用低层级的技能，形成层次化的能力体系。例如"月末结账"这个技能可以编排"应收核销"、"费用计提"、"汇率调整"等子技能，每个子技能又各自编排多个工具调用。这种层次化使得AI在面对复杂业务场景时，不必从最底层的API调用开始推理，而是站在合适的抽象层级上决策。

技能还应包含领域约束，即"什么不能做"。例如已审批的单据不允许反向修改金额，跨法人实体的资金调拨必须经过合规审查。这些约束如果不以技能的形式显式告知AI，AI可能会尝试违规操作，导致数据错误或合规风险。将约束嵌入技能描述中，比依赖工具层面的校验更可靠，因为AI在规划阶段就能规避不合法的路径。

MCP 本身没有直接的SKILL概念，目前提供的 prompts 机制并不像SKILL那样向 AI 暴露，也没有SKILL那样“渐进披露”，SKILL除了主文档，还可以包含额外的模板，脚本等，而目前 MCP prompts 是没有这个设计的。

### **行动**

有了感知和技能，AI已经知道环境中有什么、该怎么做，最后一步是真正执行——即行动。回到宜家的比喻，行动就是你拿起电钻，拧紧螺丝，把板材固定到一起。映射到AI与企业软件上，行动就是AI通过调用工具，对系统产生实际的、有副作用的变更。

行动在 MCP 上对应的是 tools。每个 tool 代表一个原子化的操作能力：创建记录、更新状态、触发流程、调用外部服务等。工具的设计需要遵循几个原则：

- 明确的副作用声明。AI 需要知道哪些工具是只读的（查询、搜索），哪些工具会修改系统状态（创建、删除、审批）。MCP 的工具定义中通过 annotations 标注工具是否具有破坏性（destructiveHint）、是否只读（readOnlyHint），AI 据此在执行前进行风险评估，避免在不恰当的时机执行不可逆操作。

- 幂等性与事务安全。企业软件中许多操作涉及资金、库存等关键数据，工具设计时应考虑重复调用的安全性。如果AI因网络超时重试了一次"扣减库存"的调用，系统不应重复扣减。这不是AI层面能解决的问题，而是工具实现层面必须保障的，但在工具描述中应告知AI哪些操作是幂等的、哪些需要先检查状态再执行。

- 权限与审批的衔接。并非所有行动都应即时生效。有些操作超出当前用户的权限，或超过了某个阈值需要人工审批。工具在这种场景下不应直接拒绝，而是返回一个"待审批"的中间状态，让AI能够告知用户"操作已提交，等待主管审批"，而不是简单报错。这要求工具的返回结构能够表达异步状态和后续动作的引导信息。

- 操作的可确认性。对于高风险操作（删除数据、大额转账、不可逆的状态变更），工具应支持"预览-确认"的两步模式。AI 先调用预览接口展示操作的影响范围，经用户确认后再执行实际变更。MCP 的 tools 本身不强制这种模式，但软件在设计工具时应将其作为最佳实践，避免AI在自动化执行链中造成不可挽回的后果。

行动还涉及跨系统的协作执行。当一个业务流程横跨多个模块甚至多个独立系统时，AI需要协调多个 MCP Server 的工具调用。例如"供应商付款"可能需要先在采购模块确认收货，再在财务模块创建付款单，最后在银行接口发起转账。A2A 协议在此场景下发挥作用，它允许不同系统的Agent之间进行结构化的任务委托和状态同步，使得跨系统的行动不再依赖单一Agent掌握所有工具，而是通过Agent间的协作完成端到端的业务流程。

此项目并不是一个代码项目，而是一个架构研讨的项目。
