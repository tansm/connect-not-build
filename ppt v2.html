领域能力层

真正的资产，
不是协议代码。

订单、审批、预算、库存、会议、知识库，这些能力才是长期价值。MCP、Skills、CLI、A2A 只是连接这些能力的不同入口。

AI 进入企业软件的三道门
◉感知Perception
≈技能Skills
✦行动Actions
01 · 感知

AI 先要知道：
我现在 在哪里？

模块、知识库、权限都重要，但更关键的是“当前”信息：当前聊天、当前项目、当前页面、当前用户。

Resources

把环境和状态暴露给 AI 主机。

Instructions

让系统介绍自己的边界与规则。

Current Context

让“当前”随用户行为实时切换。

Read Tools

把资源转成 AI 可以直接阅读的信息。

1
识别目标不是调用一个接口，而是完成一件业务上的事。
2
编排步骤校验、填充、审批、通知，形成标准流程。
3
写入约束把不能做的事提前告诉 AI，而不是等报错。
02 · 技能

工具会做事。
技能懂业务。

Skill 让 AI 站在正确的抽象层级上思考，而不是从底层工具开始猜每一步。

03 · 行动

最后一步，才是
改变系统。

副作用让 AI 明确知道哪些操作会真实改动系统。
幂等性避免因重试而重复提交、重复扣减。
审批流高风险或超阈值操作进入待审批状态。
预览确认高风险动作先展示影响，再真正执行。
实施路径

不要一次重构世界，
先连接一个闭环。

只要一个真实业务场景先跑通，后面的治理、迁移和标准化就有了现实牵引力。

A
识别核心能力把感知、技能、行动先拆清楚。
B
建立能力模型让业务核心与协议适配分离。
C
验证一个平台先让一个主流 AI 完成端到端任务。
D
沉淀治理基线认证、授权、审计、脱敏与多租户。
已知风险

标准会变，
架构不要被标准绑架。

MCP 的提示词压力

当连接很多 Server 时，工具描述可能侵占上下文窗口。

A2A 仍在早期

跨 Agent 协作方向成立，但生态还不稳定。

我们的应对

保持能力模型稳定，让协议层保持轻薄且可替换。

核心原则

能力是资产，协议只是入口。

最终判断

下一代企业 AI，
不是一个入口，
而是一组可连接的能力。

把通用能力交给快速演进的平台，把领域能力沉淀成开放标准下的资产，企业软件才能真正跟上 AI 的速度。
